Introduction

RAG-LLM Chatbot System Design & Feature Specification FastAPI · PostgreSQL · pgvector · Hugging Face LLM

1. Overview

This document specifies the complete feature set for the RAG-LLM chatbot platform and the workflow behind each feature. It is organized into six functional areas: authentication and account management, conversation management, messaging, file and document handling, retrieval-augmented generation, and platform operations. Every feature is mapped to the underlying PostgreSQL schema, so this document also serves as the bridge between the database design and the FastAPI implementation.

The system's core loop is: a user authenticates, starts or resumes a conversation, optionally attaches documents for the assistant to reference, exchanges messages with the LLM, and the platform tracks provenance (which model, which retrieved chunks, how many tokens) for every response.

2. Authentication & Account Management

2.1 Why JWT Instead of Server-Side Sessions This is a deliberate architectural decision worth explaining on its own, since it shapes the entire auth module and the user_sessions table. A traditional server-side session works by generating a random session ID at login, storing it in the database (or an in-memory store like Redis), and handing that ID to the client as a cookie. Every subsequent request requires the server to look up that ID against the store to confirm who is authenticated. This is simple to

reason about, and revocation is trivial — deleting the row instantly invalidates the session. A pure JWT (JSON Web Token) approach works differently: at login, the server issues a signed token containing the user's identity and claims directly inside the token itself (user ID, expiry, etc.). Because the token is cryptographically signed, the server can verify it on every request without touching the database at all — verification is just signature checking, which is fast and requires no round trip to Postgres. For a chatbot

backend serving many small requests per conversation (send message, load history, poll status, stream tokens), avoiding a database lookup on every single authenticated request is a meaningful performance win. It also decouples authentication from the database, which matters if the system is ever split into multiple services, since any service can verify a JWT independently without querying a shared session store. The cost of pure JWT is that it has no built-in revocation. Once issued, a token remains valid until it expires,

with no way for the server to force it invalid early — there is no row to delete. This means logout does not really work in a pure-JWT system, and a leaked long-lived token stays dangerous until it naturally expires. The system resolves this with a hybrid model, which is why the schema has a user_sessions table at all. Two tokens are issued at login: a short-lived JWT access token (minutes to roughly an hour), used on the majority

of requests exactly like pure JWT, with no database lookup; and a long-lived refresh token, which is a random opaque string stored server-side in user_sessions. The refresh token is checked against the database only when the access token expires — occasionally, not on every request — at which point the server confirms the session is not revoked and not past its expiry before issuing a new access token. This hybrid gives the system JWT's main advantage, avoiding a database hit on nearly every request, while

still supporting real logout, "log out of all devices," and session listing, since is_revoked can be flipped on the stored refresh token at any time. The is_revoked, device, and ip_address columns exist specifically to support this: they let a user see and manage active sessions, and let the server kill one specific session without logging the user out everywhere. A pure server-side session model would be simpler to reason about and

would be a reasonable choice for a smaller, single-service application, but the hybrid was chosen here because it scales better under load while preserving the control that pure JWT gives up. 2.2 Features 2.2 Sign Up Registers a new user account with a hashed password. Workflow
